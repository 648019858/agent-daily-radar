三、最佳工作流

总览：Director → Manager → Team 模式

将一人公司模拟为完整软件团队：你扮演总监制定方向，Claude Code 自动化工程经理分解任务，多个子代理并行执行，最后由自动化QA/安全审查把关。

环节 1：你写 Brief（总监）

🔗 工具：Claude Code 内建对话 + CLAUDE.md 项目文件

📋 适用范围：所有项目，是整个工作流的入口。适用于新功能开发、Bug 修复、重构等任何需要编码的场景。

维度分析
✅ 优势零学习成本，用自然语言描述需求。CLAUDE.md 提供持久化的项目上下文（架构、技术栈、规范），Claude Code 能精准理解项目全貌。
❌ 劣势描述模糊时 AI 可能跑偏。需求越复杂，Brief 需要越详细——你需要有清晰的产品思维。没有"验收标准"的习惯会导致返工。

环节 2：/batch 分解任务（工程经理）

🔗 来源：obra/superpowers 内置命令

📋 适用范围：中大型功能开发（涉及 3+ 文件的改动）。小型 Bug 修复或单文件改动不需要 batch，直接用 Claude Code 对话即可。

维度分析
✅ 优势自动将复杂任务拆分成 5-30 个并行单元，每个单元在独立 git worktree 中开发，互不干扰。大幅缩短大型功能的总开发时间，理论加速比 = 并行任务数。
❌ 劣势任务拆分质量取决于 AI 判断——拆分过细会导致合并冲突，拆分过粗则并行收益低。需要足够磁盘空间（每个 worktree 一份代码副本）。不适合强依赖关系的任务链。

环节 3：子代理并行开发（团队）

🔗 来源：obra/superpowers 的 execute-plan 子代理系统

📋 适用范围：已被 /batch 拆分好的独立任务单元。每个子代理在隔离的 git worktree 中编码、测试、提交、创建 PR。

维度分析
✅ 优势真正的并行——多个 Claude Code 实例同时工作，每个专注一个子任务。自动 commit + 自动创建 PR，无需人工管理。支持结合 Context7 MCP 获取实时文档。
❌ 劣势大量并发调用可能触发 API 频率限制（建议使用企业版 API Key）。多个子代理可能产生风格不一致的代码。需要事后统一审查来避免碎片化。

环节 4：/simplify 审查输出（高级工程师）

🔗 来源：Claude Code 内置 skill，输入 /simplify 即可触发

📋 适用范围：所有代码改动完成后。特别适合多人/多代理协作后的代码整合阶段，以及重构前的代码审计。

维度分析
✅ 优势3 个并行审查代理：Reuse Checker（发现可复用的现有代码）、Quality Checker（检查代码质量和潜在 Bug）、Efficiency Checker（发现性能瓶颈和冗余）。一次运行覆盖三大维度。
❌ 劣势审查结果的质量依赖于 AI 对代码库的理解深度——大型代码库中可能遗漏跨文件的隐含依赖。建议人工复核关键路径的审查建议。

环节 5：/code-review 最终把关（QA/安全）

🔗 来源：Claude Code 内置 skill，输入 /code-review 即可触发

📋 适用范围：PR 合并前的最终检查。建议对每个 PR 都运行，尤其是涉及安全敏感（认证、支付、数据）或核心业务逻辑的改动。

维度分析
✅ 优势4 个独立审查员从不同角度审查（安全性、正确性、可维护性、可读性），每个给出 0-100 置信度评分，只有超过阈值的问题才会被报告——减少噪音。
❌ 劣势审查覆盖度受限于静态分析，无法检测运行时行为问题（内存泄漏、并发竞态）。安全审查不能替代专业安全审计工具（如 Snyk、SonarQube）。

📊 五环节优劣势总览

环节工具最适用场景不适用场景关键风险
Brief CLAUDE.md 所有项目 — 需求模糊导致返工
/batch superpowers 大型功能（3+ 文件） 单文件小改动 拆分不当 / 合并冲突
子代理 superpowers 可并行的独立任务 强依赖链任务 代码风格不一致 / API 限流
/simplify Claude Code 内置 代码整合 / 重构前审计 探索性原型阶段 遗漏跨文件依赖
/code-review Claude Code 内置 安全敏感 / 核心逻辑 纯文档/配置改动 无法检测运行时问题
